Architecture Decision Records (ADRs)

  • ADR-0001: Frontend-First Canvas mit optionaler Helper-Runtime
    Entscheidung: UI läuft komplett client-seitig, Austausch über standardisiertes JSON-Format. Generierung erfolgt über schlanke API-Runtime. (Status: Accepted)
  • ADR-0002: Offline- & On-Premise-Fokus
    Entscheidung: Alle Skripte/Stile lokal; voller Support für on-premise LLMs (z.B. Ollama) via anpassbarer Base-URLs. (Status: Accepted)
  • ADR-0003: Standardisierte YAML & JSON-Serialisierung
    Entscheidung: Generierung von echtem Haystack-v2 YAML für deklarative Deployments ohne Python-Verbindungs-Boilerplate. (Status: Accepted)
  • ADR-0004: Zeitfenster-basierter Fehler-Korrelator
    Entscheidung: Filterung von Logs außerhalb definierter Testzeiträume vor LLM-Klassifizierung zur Token-Schonung. (Status: Accepted)

Was wird für das Deployment benötigt?

Um eine mit diesem Builder modellierte Pipeline produktiv (z.B. on-premise oder in geschlossenen VPCs) zu deployen, werden genau drei Dinge benötigt:

  • Pipeline Definition & Custom Components: Die hier generierten Python-Dateien (z.B. pipeline_components.py und pipeline_pipeline.py).
  • Standardisierte Bibliotheken: Die Core-Library haystack-ai (v2.x) sowie etwaige Zusatzpakete (z.B. openai für LLMs, pandas für CSV-Mengen).
  • Eine API-Laufzeitumgebung (Runtime Engine): Ein kleiner REST- oder Message-Broker-Wrapper (z.B. mit FastAPI), der die Pipeline lädt, eingehende Trigger empfängt, sie ausführt und die Resultate zurückgibt.

Betrieb ohne externe Cloud-Dienste (Strict On-Premise)

Diese Applikation ist so konzipiert, dass alle Komponenten offline lauffähig sind. Anstelle von OpenAI können Sie für die LLM-Verarbeitung (z.B. OpenAIGenerator) lokale, on-premise gehostete Modelle über eine OpenAI-kompatible Schnittstelle ansprechen (z.B. über LocalAI, vLLM oder Ollama). Hinterlegen Sie einfach deren lokale IP-Adresse (z.B. http://ollama-service.internal:11434/v1) in den globalen Einstellungen.

Standardisierte YAML- und JSON-Serialisierung

Wussten Sie schon? Haystack v2 verfügt über eine native, standardisierte Serialisierungs-Schnittstelle. Jede Pipeline kann komplett deklariert in YAML oder JSON abgespeichert und von jeder Haystack-Laufzeit geladen werden: 

# Pipeline in Standard-YAML exportieren (alles deklariert und serialisiert)
yaml_definition = pipeline.dumps()
with open("production_pipeline.yaml", "w") as f:
    f.write(yaml_definition)

# An einem anderen Server oder Container wieder standardisiert laden:
from haystack import Pipeline
loaded_pipeline = Pipeline.loads(yaml_definition)

Dadurch entfällt das Schreiben von Python-Instanziierungscode im Zielsystem vollständig! Sie übergeben dem standardisierten Haystack-Container einfach nur die YAML-Spezifikation per ConfigMap oder Datei-Volume.

🚀 Produktionsreife & Best Practices

Um Ihre visuell gestalteten Pipelines sicher, performant und zuverlässig in einer On-Premise-Produktionsumgebung zu betreiben, sollten Sie folgende Richtlinien befolgen:

🔒 1. Secrets & Umgebungsvariablen

Hardcoden Sie niemals API-Keys in den Komponenten-Parametern. Nutzen Sie stattdessen Umgebungsvariablen wie ${OPENAI_API_KEY}. Die Runtime liest diese automatisch via os.environ.get() aus und verhindert so Credentials-Leaks in Git-Systemen.

💾 2. Ressourcenlimitierung & OOM

Lasttest-Logs können Gigabytes groß sein. Begrenzen Sie die Zeilenanzahl über den Parameter max_lines des loaders, oder nutzen Sie die Filter-Knoten (TimeWindowFilter & KeywordFilter) direkt nach dem Laden, um den Arbeitsspeicher zu schonen.

⚡ 3. Asynchrone Jobs (Task Queues)

Für umfangreiche Log-Analysen, die länger als 10-15 Sekunden dauern, ist ein synchroner HTTP REST-Call ungeeignet. Lagern Sie die Pipeline-Ausführung in Hintergrund-Arbeiter aus (z.B. Celery, Redis Queue oder FastAPI BackgroundTasks) und nutzen Sie Webhooks zur Resultat-Übermittlung.

📈 4. Monitoring & Logging

Überwachen Sie die Pipeline-Laufzeiten. FastAPI bietet hervorragende Integrationen für Prometheus-Metriken. Protokollieren Sie jeden Schritt mit standardisiertem Python-Logging, um bei LLM-Fehlern oder API-Timeouts sofort benachrichtigt zu werden.

🔧 Troubleshooting (Problemlösungen im On-Premise-Betrieb)

  • Frage: Warum erhalte ich "Connection Refused" beim Aufruf von Ollama aus dem Container?
    Lösung: Im Docker-Netzwerk verweist localhost auf den Container selbst. Nutzen Sie http://host.docker.internal:11434/v1 (Windows/macOS) oder die Server-IP, und stellen Sie sicher, dass Ollama auf dem Host mit OLLAMA_HOST=0.0.0.0 gestartet wurde.
  • Frage: Wie gehe ich mit UnicodeDecodeError beim Parsen von Logs um?
    Lösung: Lasttest-Dateien enthalten manchmal binäre Artefakte oder seltene Zeichensätze. Unsere Loader unterstützen den Parameter encoding. Setzen Sie diesen auf utf-8 oder als Fallback auf latin-1.
  • Frage: Warum bricht die Pipeline mit "Missing required input" ab?
    Lösung: Überprüfen Sie, ob Sie im Canvas alle Pflicht-Schnittstellen (gekennzeichnet mit Stern) verbunden haben. Sie können auch beim Starten der Pipeline im REST-Trigger Standardwerte im inputs-JSON mitgeben.
